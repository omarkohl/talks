Design Pattern

“A design pattern is the re-usable form of a solution to a design problem.”

Wikipedia

Design Pattern

Idea introduced by architect Christopher Alexander in A Pattern Language (1977)

Software Design Pattern

Popularized by Design Patterns: Elements of Reusable Object-Oriented Software (1994) by Erich Gamma, Richard Helm, Ralph Johnson and John Vlissides (a.k.a. GoF, Go4, Gang of Four)

Software Design Pattern

  • reusable solution to a commonly occurring problem within a given context in software design
  • formalized best practices that the programmer can use
  • intermediate [approach] between the levels of a programming paradigm and a concrete algorithm
  • some patterns can be rendered unnecessary in languages that have built-in support for solving the problem they are trying to solve

Wikipedia

Software Design Pattern

  • Best known DPs apply to OOP but not limited to it
  • Design and patterns are NOT independent from the underlying technology (wood vs. bricks)
  • Need a name
  • Known uses have to be documented (DPs are discovered!)
  • Some classic DPs are workarounds for static typing
  • Categories: Creational, Structural, Behavioral, Concurrency

Small procedural patterns (by Brandon Rhodes)

Example 1

GoF book says (intro):

  • Program to an interface not an implementation
  • Favor object composition over inheritance

Example 2

Factory Method

Create objects without having to specify the exact class of the object that will be created

Example 3

Singleton

Restrict the instantiation of a class to one object

Example 4

Adapter

The Adapter design pattern allows otherwise incompatible classes to work together by converting the interface of one class into an interface expected by the clients.

Example 5

Facade

Provide a unified interface to a set of interfaces in a subsystem. Facade defines a higher-level interface that makes the subsystem easier to use.

Example 6

Bridge

Meant to decouple an abstraction from its implementation so that the two can vary independently (don't subclass for several purposes)

Example 7

Iterator

Provide a way to access the elements of an aggregate object sequentially without exposing its underlying representation.

Example 8

Strategy

Enables an algorithm's behavior to be selected at runtime

Example 9

Template

Define the skeleton of an algorithm in an operation, deferring some steps to subclasses. Template method lets subclasses redefine certain steps of an algorithm without changing the algorithm's structure.
